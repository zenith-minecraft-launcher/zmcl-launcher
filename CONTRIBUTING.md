# 贡献准则

感谢你有兴趣为 Zenith 贡献代码或文档！为了让代码评审更高效、项目维护更可持续，请在提交 Pull Request 之前阅读以下内容。

---

## 开始之前

1. **请先搜索现有 Issue 与 PR**，避免重复工作。如果你想修复的 Bug 或实现的功能已有相关讨论，请在其中说明你打算接手，避免与他人撞车。
2. **重大改动请先通过 Issue 或社区交流讨论**，说明你的设计思路与实现方案，得到维护者认可后再动手开发，避免花了很多精力却无法合并。
3. **小改动可以直接提交 PR**，例如：修复错别字、修正明显的逻辑错误、微调 UI 样式等。

## 开发环境

本项目基于 Electron，构建工具为 electron-builder。

- **运行开发版**：`npm run dev`
- **本地调试**：`npm start`
- **构建发行包**：`npm run build`（可按需使用 `:win`、`:mac`、`:linux`）

请确保你的 Node.js 版本与项目兼容，并在提交改动前完成一次完整的本地构建，确认启动流程、游戏启动、登录、模组下载等核心路径均无异常。

## 分支与提交

- 请从主干分支（通常为 `main` 或 `master`）切出你的开发分支。
- 分支命名建议：`feature/xxx`、`fix/xxx`、`refactor/xxx`、`docs/xxx`。
- 提交信息请使用有意义的描述，避免 `fix`、`update` 这种难以回溯的信息。建议遵循 Conventional Commits 风格，例如：
  - `fix(launch): 修复 Java 路径包含空格时参数解析失败`
  - `feat(auth): 新增自定义第三方验证服务器支持`
  - `refactor(download): 重写模组下载队列逻辑`

## 代码风格

- 遵循项目既有的代码风格与缩进规范；新增文件应与周围文件保持一致。
- 命名清晰可读，变量名和函数名应能表达其含义。
- 避免引入不必要的依赖；若必须引入新的 npm 包，请在 PR 描述中说明其必要性和安全评估。
- 涉及敏感信息（账号、Token、密码等）**绝对不能**出现在代码或日志中。
- 涉及文件系统、网络请求、进程调用的操作，需做好异常处理，避免静默失败或未捕获的异常导致启动器卡死。

## Pull Request 规范

1. **一个 PR 只做一件事**。如果你同时修改了多个模块，请拆分为多个 PR 分别提交。
2. **PR 标题简明扼要**，描述中请说明：
   - 你做了什么改动；
   - 为什么这样改（解决了什么问题 / 实现了什么功能）；
   - 如何验证（测试步骤、截图、日志等）；
   - 是否存在 Breaking Change。
3. **请保持 PR 的 commit 历史干净**。必要时可以使用 `rebase` 或 `squash` 整理 commit。
4. **请在提交前进行自测**，确保改动在你的环境中可以正常工作，并说明测试覆盖的场景。
5. **PR 合并前请保持响应**。维护者可能会在评审中提出修改意见，请及时跟进与回复；长期无响应的 PR 可能会被关闭。

## 资源与二进制文件

- 请勿在仓库中提交较大的二进制文件、压缩包或外部资源，除非它们是构建必须且无法通过网络获取的。
- 图标、图片等静态资源请放置在 `resources/` 或 `src/renderer/assets/` 目录下，并注意压缩优化。
- Java、模组、版本文件等运行时资源不应随源码提交。

## 安全与合规

- 所有贡献代码应遵循项目的开源许可（GPL-3.0），不得引入与该许可冲突的内容。
- 禁止在代码中硬编码任何形式的凭据、密钥、私有链接或个人信息。
- 涉及用户账号、认证与网络请求的部分，应采用最小权限与加密传输原则。
- 不得包含对用户系统产生副作用（植入、捆绑非必要软件、修改系统关键项等）的逻辑。

## 翻译与文档

- 若你贡献翻译或多语言支持，请在 PR 中说明新增/修改的语言项，并确保关键文案翻译准确。
- 文档修改（包括本文件）应与代码改动同步更新。

## 评审与合并

- 维护者会尽可能在合理时间内完成评审，具体时间视改动规模与维护精力而定。
- 评审意见不是否定，请以合作心态讨论技术方案。请在交流中保持礼貌与耐心。
- 合并前可能会要求你再做几轮调整，这是正常流程，并不代表你的贡献不被欢迎。

---

再次感谢你的贡献！每一个有价值的 Issue 和 PR 都在让 Zenith 变得更好。
